## 回報安全漏洞

如果您發現安全漏洞，請**不要**透過公開的 Issue 回報。

### 回報方式

請透過以下方式私下回報安全問題：

1. 發送電子郵件至：yang06140425@gmail.com
2. 包含以下資訊：
   - 漏洞類型
   - 受影響的檔案路徑
   - 重現步驟
   - 潛在影響評估
   - （如有）建議的修復方式

### 回應時間

- 我們將在 **48 小時內** 確認收到您的回報
- 我們將在 **7 天內** 提供初步評估
- 我們將在 **30 天內** 發布修復（視漏洞嚴重程度調整）

## 安全最佳實踐

### 環境變數管理

```bash
# 正確做法：使用 .env 檔案（不提交至版本控制）
cp .env.example .env
# 編輯 .env 填入實際金鑰

# 錯誤做法：直接在程式碼中硬編碼金鑰
# api_key = "sk-xxxxxxxxxxxx"  # 絕對不要這樣做！
```

### API 金鑰安全

1. **定期輪換**：建議每 90 天輪換一次 API 金鑰
2. **最小權限**：只授予必要的 API 權限
3. **監控使用**：定期檢查 API 使用量是否異常
4. **分離環境**：開發、測試、生產環境使用不同的金鑰

### 敏感資料處理

- 查詢結果中可能包含法規原文，請依據適用法律處理
- 使用者查詢歷史應妥善保護
- 定期清理不必要的快取和歷史資料

## 已知安全考量

### 網路爬蟲

本系統使用網路爬蟲取得公開法規資料，請注意：

- 遵守目標網站的 `robots.txt`
- 設定合理的請求間隔，避免對目標網站造成負擔
- 僅用於合法用途

### LLM 安全

使用大型語言模型時的注意事項：

- 輸入驗證：系統會對使用者輸入進行基本驗證
- Prompt Injection：系統 prompt 與使用者輸入已分離處理
- 輸出過濾：LLM 輸出會經過格式驗證
